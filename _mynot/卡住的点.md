这一个 story 只完成了最小底座：
  证明“屏幕观察 → 决策 → 执行动作 → 再观察”这条链路的规则是对的。

  它完成的是：

  scene-reader 读屏幕
  → actor 做决定
  → action-bridge 执行动作
  → scene-reader 再读新屏幕

  并且锁住了几个反例：

  - server 不能冒充 agent 写消息
  - browser 不能用 /api/* 当进度证明
  - action-bridge 不能自己做策略
  - scripted screen 不能冒充真实观察

  但它还没有完成完整 CDDA demo，例如还缺：

  1. 真的连接 CDDA 游戏进程
  2. 真的有 scene-reader agent 从屏幕/tmux 读信息
  3. 真的有 actor agent 根据观察做下一步决策
  4. 真的有 action-bridge 把动作发进游戏
  5. browser 只展示订阅结果，而不是调用 /api/reach-playable
  6. 清理旧的 reach_playable()、/api/state、server-side shortcut 残留
  7. human input 正确走 human_input channel
  8. commit / restore / audit view 等后续能力

  所以更准确地说：

  > 这一个 story 完成了 CDDA demo 的“可信消息链底座”，不是完成整个 CDDA demo。

  为什么 PROGRESS.md 里只有一条 story？
  因为这个工作流规定 一次只推进一条 story，防止又把所有东西混在一起做，导致之前那种 server 偷偷代写、browser 偷偷调 API 的问题重新出现。

  下一步应该开第二条 story，例如：

  S001-live-scene-reader-runtime

  目标可以是：让真实 scene-reader agent 从 CDDA 屏幕读取信息，并产出符合 S000 contract 的 scene_observation。
